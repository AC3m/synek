# Implementation Plan: Junction Garmin PoC Integration

**Branch**: `011-junction-garmin-poc` | **Date**: 2026-03-15 | **Spec**: [spec.md](./spec.md)

## Summary

Add a temporary, fully-isolated Garmin connection capability to the Integrations tab using Junction's wearable API. A user clicks "Connect Garmin", a backend Edge Function creates a Junction user and returns a short-lived link token, and the Junction popup widget handles Garmin credential entry. Incoming workout webhooks are verified via Svix HMAC and stored as raw JSONB in two isolated `junction_poc_*` tables that have no dependencies on existing data models. All new code lives in new files; the only modification to existing files is one added child component in `IntegrationsTab.tsx`. Every artefact is designed for clean, traceless removal after the PoC concludes.

## Technical Context

**Language/Version**: TypeScript 5 (strict) — React 19 frontend, Deno Edge Functions
**Primary Dependencies**: `@tryvital/vital-link` (1.7 KB gzipped, new), `svix` via esm.sh (Deno, Edge Function only — not a frontend bundle concern)
**Storage**: Supabase PostgreSQL — two new isolated tables (`junction_poc_connections`, `junction_poc_events`)
**Testing**: Vitest + @testing-library/react; mock implementations for all query functions
**Target Platform**: Web SPA (React Router 7, SPA mode) + Supabase Edge Functions (Deno)
**Performance Goals**: Connect flow completes in under 3 minutes end-to-end; webhook storage is best-effort within Supabase Edge Function execution limits
**Constraints**: `@tryvital/vital-link` is 1.7 KB gzipped — within the 50 KB Constitution threshold. React 19 peer dep is `>=16`; risk accepted (tiny surface area, CDN-loaded widget runtime).
**Scale/Scope**: Single developer PoC; one real Garmin account; Junction sandbox API key

## Implementation Phases

### Phase 1 — Database & Backend (no frontend yet)

**Goal**: Webhook pipeline working end-to-end, testable from Junction dashboard.

**Steps**:

1. **Migration** — create `junction_poc_connections` and `junction_poc_events` tables with RLS policies (see `data-model.md`)
2. **`junction-create-user` Edge Function** — creates Junction user + generates link token; secured by Supabase JWT auth
3. **`junction-webhook` Edge Function** — verifies Svix signature, deduplicates by `svix-id`, stores raw payload; public endpoint
4. **Env secrets** — add `JUNCTION_API_KEY`, `JUNCTION_API_BASE_URL`, `JUNCTION_WEBHOOK_SECRET` to Supabase project secrets
5. **Register webhook URL** in Junction sandbox dashboard: `https://<project>.supabase.co/functions/v1/junction-webhook`

**Verification**: Use Junction sandbox dashboard to send a test event → confirm row appears in `junction_poc_events`.

---

### Phase 2 — Query Layer & Hook

**Goal**: Frontend data layer ready; mock mode works.

**Steps**:

1. **`app/types/junction-poc.ts`** — `JunctionPocConnection`, `JunctionPocEvent` types
2. **`app/lib/mock-data/junction-poc.ts`** — in-memory mock connection state with `resetMockJunctionPoc()`
3. **`app/lib/queries/junction-poc.ts`**:
   - `fetchJunctionConnection(userId)` — reads `junction_poc_connections`; mock returns null or a fixture
   - `createJunctionConnection(appUserId, junctionUserId)` — inserts connection row
   - `disconnectJunctionConnection(appUserId)` — updates `status = 'disconnected'`
   - Row mapper `toJunctionConnection()` for snake_case → camelCase
4. **`app/lib/hooks/useJunctionConnection.ts`**:
   - `useJunctionConnectionStatus(userId)` — query hook, `queryKey: ['junction-poc', 'connection', userId]`
   - `useJunctionConnect()` — mutation: calls `junction-create-user` Edge Function, then stores connection in DB via `createJunctionConnection`; implements optimistic update
   - `useJunctionDisconnect()` — mutation: calls `junction-disconnect` Edge Function, then updates DB; optimistic: sets status to `disconnected` immediately
5. **Query key** added to `app/lib/queries/keys.ts`:
   ```typescript
   export const junctionPocKeys = {
     all: ['junction-poc'] as const,
     connection: (userId: string) => [...junctionPocKeys.all, 'connection', userId] as const,
   }
   ```

---

### Phase 3 — UI Component & Integration

**Goal**: User can connect/disconnect Garmin from the Integrations tab.

**Steps**:

1. **`app/components/settings/JunctionGarminSection.tsx`**:
   - Named export, `className?: string` prop
   - Uses `useJunctionConnectionStatus` + `useJunctionConnect` + `useJunctionDisconnect`
   - **Disconnected state**: "Connect Garmin" `Button` — on click, calls `junction-create-user` Edge Function, then calls `open(linkToken)` from `useVitalLink`
   - **Connected state**: `Badge` showing "Garmin connected" + "Disconnect" `Button` with confirmation
   - `useVitalLink` config: `onSuccess` triggers `useJunctionConnect` mutation to store the connection; `onExit` resets loading state; `onError` shows error toast
   - Install: `pnpm add @tryvital/vital-link`

2. **`app/components/settings/IntegrationsTab.tsx`** — add at the bottom of the rendered output:
   ```tsx
   <JunctionGarminSection className="mt-6" />
   ```
   No other changes to this file.

3. **i18n** — add `junction.*` keys to `en/common.json` and `pl/common.json`:
   ```json
   "junction": {
     "connectButton": "Connect Garmin",
     "connectedBadge": "Garmin connected",
     "disconnectButton": "Disconnect",
     "disconnectConfirm": "Are you sure you want to disconnect your Garmin account?",
     "connecting": "Connecting...",
     "errorLoad": "Failed to load the connection widget. Please try again.",
     "errorConnect": "Failed to connect Garmin. Please try again.",
     "errorDisconnect": "Failed to disconnect. Please try again."
   }
   ```

### Phase 4 — End-to-End Verification

**Goal**: Real Garmin workout received and inspectable.

**Steps**:

1. Log in with a real app account
2. Navigate to Settings → Integrations
3. Click "Connect Garmin" — Junction popup opens
4. Enter real Garmin credentials — popup closes, "Garmin connected" badge appears
5. Complete a real Garmin workout and sync via the Garmin app
6. Check `junction_poc_events` table in Supabase dashboard for `daily.data.workouts.created` event
7. Inspect the raw `payload` JSONB — confirm workout data fields are present and correct
